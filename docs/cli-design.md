# CLI 设计规范

本文档定义 `urn` 命令行的输出与交互约定。后续新增命令、调整现有命令，默认都应遵循这里的规则。

## 目标

- 默认输出优先服务人类阅读
- 结构化输出通过显式参数获取
- 表格输出必须正确处理 CJK 宽字符
- 不同命令的输出风格保持一致，避免一条命令一个习惯
- 输出设计要能稳定测试，避免只凭肉眼判断

## 总原则

### 默认面向人

CLI 的默认输出是给人在终端里直接看的，不是给脚本消费的。

这意味着：

- 默认不要输出 JSON
- 默认格式要能快速扫读
- 默认格式要适配中英混排
- 默认格式要尽量控制噪音，不把底层原始结构直接倒出来

如果调用方是 AI Agent、shell 脚本、外部程序，它们应显式传入 `--format`。

### 机器显式选择结构化格式

所有适合机器消费的结构化输出，都应通过显式参数启用。

推荐约定：

- `--format json`
- `--format jsonl`
- `--format csv`
- `--format tsv`

不要依赖默认输出做程序解析。

### 格式能力按命令语义开放

不是所有命令都要支持所有格式。

- 行记录查询类命令适合支持 `json/jsonl/csv/tsv/table`
- 聚合统计类命令通常只需要 `json/table`
- 摘要类命令通常只需要 `json/text`
- 执行反馈类命令通常只需要 `json/text`

不要为了“统一”强行给不适合的命令加 `csv`。

## 输出格式定义

### table

`table` 是默认的人类友好表格格式，必须满足：

- 按显示宽度对齐，不按字符串长度对齐
- 正确处理 CJK 宽字符
- 能处理中英混排
- 空值展示规则统一
- 超长内容要截断，避免破坏终端布局

`table` 不是简单的 tab 分隔，也不是伪 TSV。

### text

`text` 用于摘要、执行结果、概览等不适合表格的场景，要求：

- 一眼能看到最重要结论
- 先结论，后细节
- 字段数量控制在必要范围内
- 保持稳定结构，便于测试

适用场景：

- `summary`
- `ingest`
- `sync`

### json

`json` 用于完整结构化输出，要求：

- 字段命名稳定
- 结构尽量显式
- 不混入展示文案
- 适合作为脚本和 Agent 的默认结构化接口

### jsonl

`jsonl` 仅用于逐行记录流式输出。适用于：

- `query`
- 导出事件记录

聚合结果不要为了“统一”输出成 `jsonl`。

### csv

`csv` 只用于天然是“行记录导出”的命令，要求：

- 表头稳定
- 字段顺序稳定
- 统一转义规则
- 换行、引号等特殊字符正确转义

适用范围应严格控制，优先用于 `query`。

### tsv

如果需要与 shell 文本工具更自然地配合，可以提供 `tsv`。但 `tsv` 不是人类友好表格，不应作为默认输出。

## CJK 表格要求

### CJK 是一等公民

表格输出必须把 CJK 当作正常输入，而不是特殊情况。

最低要求：

- 中文路径不乱列
- 中文标题不乱列
- 中英混排不乱列
- 截断后不出现明显错位

### 按显示宽度计算

列宽计算必须基于显示宽度，而不是 JavaScript 字符串长度。

至少要考虑：

- CJK 宽字符
- 全角标点
- 半角 ASCII
- 空字符串与空值

如果后续引入彩色输出，还要确保 ANSI 转义序列不参与宽度计算。

### 截断规则

需要截断的列应按显示宽度截断，并使用统一的省略形式。

建议：

- 优先截断内容列，不优先截断时间列、类型列
- 截断后追加 `…`
- 截断规则在同一命令中保持一致

### 窄终端降级

如果终端宽度不足，输出应优雅降级，而不是继续硬撑出错位表格。

允许的降级方式：

- 减少展示列
- 压缩低优先级列
- 切换为多行记录块

不允许输出明显错位但仍自称 `table`。

## 命令格式策略

当前建议如下：

### query

- 默认：`table`
- 支持：`table/json/jsonl/csv/tsv`

原因：

- 既有人工浏览场景
- 也有导出和程序消费场景

### stats

- 默认：`table`
- 支持：`table/json`

原因：

- 本质是聚合结果
- 默认应快速看分布

### summary

- 默认：`text`
- 支持：`text/json`

原因：

- 摘要比结构更重要
- 默认要先让人读懂

### ingest

- 默认：`text`
- 支持：`text/json`

原因：

- 这是执行反馈，不是数据导出

### sync

- 默认：`text`
- 支持：`text/json`

原因：

- 重点是看本次同步做了什么、有没有新增、哪些 batch 生效

### sources list

- 默认：`table`
- 支持：`table/json`

### nodes list

- 默认：`table`
- 支持：`table/json`

## 兼容性规则

### 默认输出变更属于显式决策

修改默认格式会影响现有脚本和使用习惯，因此必须：

- 在 PR 里明确说明
- 更新 README 示例
- 更新 e2e 测试

如果只是新增结构化格式，不属于高风险变更。

### 机器接口要尽量稳定

一旦某个命令支持 `--format json`、`--format jsonl` 或 `--format csv`，就要把它视为对外接口。

调整时要注意：

- 字段名不要随意改
- 字段顺序不要随意改
- 不要在结构化输出里混入展示文案

## 实现建议

### 统一输出层

不要在每个命令里散落 `console.log(JSON.stringify(...))` 和手写格式逻辑。

应抽出共享输出层，至少包含：

- 格式选择
- 视图模型到文本的转换
- `table/text/json/jsonl/csv/tsv` 渲染

### 先产出视图模型，再渲染

命令本身先整理出与语义对应的数据结构，再由 formatter 渲染。

不要让 formatter 直接理解数据库实体细节，也不要在命令里直接拼字符串。

### 少做隐式魔法

输出格式切换应显式、直接、可预期。

避免：

- 根据是否是 TTY 自动改字段
- 同一个 `--format` 在不同命令下含义漂移
- 无提示地混用表格和自由文本

## 测试要求

每次新增或修改输出格式，至少覆盖以下测试。

### 单元测试

- ASCII 对齐
- CJK 对齐
- 中英混排
- 空值展示
- 超长内容截断
- `json/jsonl/csv/tsv` 转义与稳定性

### e2e 测试

- 默认格式符合规范
- 显式 `--format` 可用
- 关键表头或关键文本稳定
- 结构化输出可被解析

测试数据里必须包含中文样本，不能只用英文数据验证“支持 CJK”。

## 文档要求

如果命令新增格式或修改默认格式，必须同步更新：

- `README.md`
- 本文档
- 相关测试

如果变更影响用户使用习惯，还应在变更说明中明确指出。

## 决策准则

遇到拿不准的情况，按下面顺序判断：

1. 默认输出是不是更适合人直接看
2. 机器是否可以通过显式 `--format` 获得稳定结构
3. 表格是否真正支持 CJK，而不是看起来像表格
4. 这个格式是否真的符合该命令的语义
5. 是否有测试和文档一起落地

如果其中任一项不能满足，不应直接合入。
